# Personal intelligence layer — architecture & plan

**Status:** design, not built. Build **after multi-user capture** (`MULTI-USER-CAPTURE.md`) —
that fills the feedback loops with data. The *why* is in `PRODUCT-THESIS.md`. This is the
*how*: turning FocusBoard's three behavioural loops into a per-user model that helps the
user, cheaply and trustworthily.

---

## 1. Architecture in one picture

```
 RAW EVENTS                DERIVED SIGNALS              INSIGHTS                 SURFACE
 (already stored)          (cheap, incremental)        (mostly deterministic)   (in context)
─────────────────         ──────────────────────      ────────────────────     ─────────────
 capture_queue    ──┐                                  estimate for THIS card   inline at
 focus_sessions   ──┼──►  user_signals  ──────────►    aging / rot flags    ──► focus-start,
 cards            ──┘     (per-user feature store)     triage ranking           triage, daily
 app_state          │                                  productive-window        brief, weekly
                    │                                   blocker clusters         review
                    ▼                                        │                        │
              suggestion_events  ◄──────────────────────────┴────────────────────────┘
              (every suggestion + the user's reaction — the next training signal)
```

**Principle: a derived-signals layer sits between raw events and any AI.** Most insight is
*statistics over the user's own data* (free, explainable, trustworthy). The LLM is reserved
for the few things only it can do (free-text → structure, natural-language synthesis) — and
it's capped (§6). That separation is what makes this a product, not a per-action wrapper.

---

## 2. The signals layer (the heart)

A per-user feature store — `user_signals` rows / materialized views, updated **incrementally**
as events land (a new `focus_session`, a triage decision, a card move). All deterministic SQL.

| Signal | Derived from | Used for |
|---|---|---|
| **Estimation bias** | `focus_sessions.planned_minutes` vs actual (`ended-started`), by tag/size | realistic per-card estimates |
| **Productive windows** | progressed/completed rate by hour-of-day / weekday | "do this in your peak window" |
| **Throughput / capacity** | sessions/day, completed cards/week | realistic daily planning |
| **Blocker patterns** | `outcome='blocked'` frequency + clustered notes | surface recurring blockers/deps |
| **Finish vs avoid** | completed vs abandoned by task type/size/time | what to schedule when |
| **Triage signal** | `capture_queue` kept/edited/dismissed by source/sender/pattern | inbox ranking, auto-confidence |
| **Card aging** | `cards` dwell time per column, archived-without-completion | rot / avoidance detection |

Each signal carries a **sample size + confidence**; nothing is surfaced below a threshold
(cold-start guard, §5).

---

## 3. Inference — deterministic first, LLM only where it must

- **Deterministic (the majority):** estimates, windows, throughput, aging, triage ranking
  are arithmetic over `user_signals`. Cheap, instant, *explainable* ("your last 12 sessions
  on `#client` averaged 70m vs 30m planned"). Explainability is a trust feature.
- **LLM (narrow, capped):**
  - personalised **parse** of a capture into cards (today's `api/capture/process.ts`, now on
    the fast-path + capped path from `MULTI-USER-CAPTURE.md` §5),
  - **clustering** free-text (blocker notes → recurring categories), run in batch,
  - **synthesis** of the daily brief into natural language (one small call/user/day).
- **Gating:** an insight emits only with enough samples and above a confidence floor —
  prevents confident nonsense early on.

---

## 4. Surfacing — in context, explainable, dismissible

Ranked by value/friction:
1. **Inline (best):** at **focus-start**, show the personalised estimate; at **triage**,
   pre-sort the inbox by learned signal and pre-set auto-add confidence.
2. **Daily brief (opt-in):** one proactive synthesis — aging cards, optimistic estimates,
   your peak window, a suggested order. Not a nag; a glance.
3. **Weekly honest mirror:** patterns over time — estimation-accuracy trend, finish-vs-avoid,
   where time actually went.

Every suggestion is **explainable** (shows the "because…") and **dismissible**; the human
stays in control. No silent auto-actions.

---

## 5. The feedback loop & cold start

- **`suggestion_events`** (`id, user_id, kind, payload, shown_at, reaction, reacted_at`) logs
  every suggestion and the user's response (accepted / edited / ignored / dismissed). This
  (a) ranks future suggestions and (b) is the labelled dataset for later model tuning — again,
  self-generated by use.
- **Cold start:** new users have no signals → fall back to the heuristic fast-path + sane
  defaults; surface insights only as each signal crosses its confidence threshold. The app is
  fully useful from day one and *gets sharper*, never broken-until-trained.

---

## 6. Cost & privacy architecture (non-negotiable)

**Cost (rule: estimate before spend):**
- The signals layer is **free** (SQL). The moat is deterministic — that's deliberate.
- LLM spend is bounded: capture parse (fast-path + cap), one daily-brief call/user/day
  (batched), occasional clustering. **Hard per-user and global budgets**; over budget,
  features degrade to deterministic/fast-path, never silently overspend.

**Privacy / trust (the product itself):**
- All derivation is **per-user, RLS-isolated**; a user's model is built only from *their*
  data. **No cross-user training** by default.
- The user can **see, export, and delete** their signals and suggestion history.
- Any future aggregate/cross-user learning is **opt-in**, aggregated, and privacy-preserving.
- "Your data trains only your model, and it's yours" is a headline feature, not fine print.

---

## 7. Phased plan (each phase shippable; each compounds on the signals layer)

- **P1 — Signals layer + estimation.** Build `user_signals` (incremental) and ship the single
  highest-value, fully-deterministic feature: **realistic per-card time estimates** at
  focus-start. Proves the loop end-to-end, zero LLM.
- **P2 — Triage personalisation.** Learn keep/dismiss signal → ranked inbox + smarter
  auto-add confidence; personalised capture parsing.
- **P3 — Honest mirror.** Card-aging/rot flags + finish-vs-avoid patterns + the weekly review.
- **P4 — Proactive brief.** Daily synthesis (capped LLM) + productive-window scheduling
  suggestions; wire `suggestion_events` and let reactions re-rank.
- **P5 — Blocker intelligence.** Cluster blocker notes → recurring blockers/dependencies;
  surface "this tends to block on X."

**Dependencies:** needs multi-user capture first (data volume + the channels that feed the
loops). Each phase ships value alone; the signals layer is the shared foundation.

---

## 8. Data model additions
```sql
user_signals(user_id, key, value jsonb, sample_n int, confidence float, updated_at)
  -- one row per (user, signal key); incrementally maintained; RLS by user_id
suggestion_events(id, user_id, kind, payload jsonb, shown_at, reaction, reacted_at)
  -- the feedback loop + future training labels; RLS by user_id
-- focus_sessions / capture_queue / cards already carry the raw events (no change needed)
```
